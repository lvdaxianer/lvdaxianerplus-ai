技术方案对比

三种实现路径，各有优劣

A

独立 MCP 服务（推荐）

✓ 我的推荐

优势

  • 职责清晰，独立演进
  • 不影响现有 mcp-http-gateway
  • 可单独部署和扩展
  • 依赖更少，更轻量

劣势

  • 需要新建项目结构
  • 配置管理需重新设计
B

嵌入 mcp-http-gateway

作为现有项目的功能模块

优势

  • 复用现有基础设施
  • 配置、日志、监控统一
  • 开发工作量较小

劣势

  • 职责混杂，耦合度高
  • mcp-http-gateway 变得臃肿
  • 不符合单一职责原则
C

独立 MCP + 共享基础设施

新项目，复用通用工具库

优势

  • 独立服务，职责清晰
  • 复用 types.ts 等通用定义
  • 配置风格保持一致

劣势

  • 需要维护共享库
  • 依赖关系增加

我的推荐理由

选择 方案 A（独立 MCP），因为架构图生成是一个独立的能力域，与 HTTP 网关职责完全不同。 独立服务更易于维护、测试和扩展，未来还可以单独迭代而不影响网关的稳定性。
