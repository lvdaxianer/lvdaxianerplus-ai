# 配置管理热更新/重启功能测试故事线

## 故事目标

- **核心问题**：用户修改配置文件时，需要明确区分哪些配置变更可以热更新生效，哪些需要重启服务，并在重启前提醒用户服务会中断
- **目标用户**：MCP HTTP Gateway 的运维人员和配置管理员
- **预期产出**：配置保存 API 正确识别变更类型，热更新变更立即生效，重启变更提前警告用户

## 关键节点/里程碑

- **起点**：用户通过 Dashboard 或 API 修改配置
- **中间步骤**：
  1. 配置比较模块检测变更类型（热更新 vs 重启）
  2. API 返回不同的响应消息和警告
  3. 热更新变更立即生效，重启变更触发服务重启
  4. 重启后新配置生效，验证数据迁移说明
- **终点**：配置变更完成，用户收到明确的反馈

## 角色/参与者

| 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 用户 | 修改配置并保存 | 新配置 JSON | 等待响应 |
| 配置比较模块 | 检测变更类型 | 旧配置 + 新配置 | 变更列表 + 是否需重启 |
| API Handler | 返回响应并触发操作 | 比较结果 | API 响应 + 重启调度 |
| 服务重启模块 | 优雅重启服务 | 重启原因 | 新进程启动 |

## 流程/路径

### 流转条件
- 配置变更类型决定处理路径

### 分支路径
- **热更新变更** → 立即应用变更 → 返回成功消息
- **重启变更** → 返回警告消息 → 调度重启 → 新进程启动

### 配置变更分类

| 配置项 | 变更类型 | 处理方式 |
|--------|---------|---------|
| `sqlite.dbPath` | 需重启 | 关闭旧连接 → 新进程初始化新数据库 |
| `transport` | 需重启 | STDIO ↔ SSE 模式切换需重启 |
| `httpPort` / `ssePort` | 需重启 | 端口变更需重启 |
| `tools` | 需重启 | 工具列表变更需重新加载 |
| `rateLimit` / `concurrency` | 需重启 | 核心组件需重启 |
| `cache` | 热更新 | 立即生效，不重启 |
| `mock` | 热更新 | 立即生效，不重启 |
| `logging.level` | 热更新 | 立即生效，不重启 |
| `circuitBreaker` | 热更新 | 立即生效，不重启 |

## 修复记录

### 问题 1：Dashboard 保存配置不显示重启警告

**问题描述**：用户通过 Dashboard 保存配置时，即使修改了需要重启的配置项（如 sqlite.dbPath），也没有显示重启警告。

**问题原因**：前端 `saveConfig()` 函数没有检查 API 响应中的 `restartScheduled` 字段。

**修复方案**：
- 更新 `scripts.ejs` 中的 `saveConfig()` 函数
- 检查 `restartScheduled` 字段，显示重启警告 Toast
- 服务重启后自动刷新页面

**修复代码位置**：`src/routes/templates/partials/scripts.ejs` 行 834-862

### 问题 2：配置保存后 SQLite 持久化数据丢失

**问题描述**：用户修改配置后刷新页面，SQLite 持久化的配置（mock、cache、fallback）丢失。

**问题原因**：
- GET /api/config 返回合并配置（文件 + SQLite）
- PUT /api/config 将合并配置全部写入文件
- SQLite 特有配置被写入文件，下次加载时覆盖 SQLite 数据

**修复方案**：
- 添加 `stripSQLiteConfigs()` 函数过滤 SQLite 特有配置
- 保存时只写入文件配置，不包含 SQLite 持久化数据
- SQLite 配置独立管理，不受文件配置影响

**修复代码位置**：`src/routes/handlers/config.handler.ts` 行 50-70
